01

Everything is
a holon

hólos + -on

Greek: “whole” + “part”

A holon is both a self-contained entity and a component of a larger system. It is whole on its own. It is also a part.

In OASIS, every data entity is a holon: users, applications, quests, NFTs, game objects. Each one knows what it is, where it belongs, and how to travel between providers without losing a single byte.

One data structure. Every backend.

Identity

Every holon has a GUID, version history, audit trail, and change detection. It knows who created it, who modified it, and when.

Hierarchy

Holons nest inside holons. A parent references children; children reference their parent. Infinite depth. One tree.

Portability

Each holon maps its ID to every provider it touches via ProviderUniqueStorageKey. Move between Ethereum, MongoDB, or Holochain with zero data loss.

02

The kernel that
never compromises

Every read and write in OASIS passes through the kernel. It enforces invariants: holons must carry valid GUIDs, parent references must resolve, provider keys must map to real storage. These rules are not optional and cannot be bypassed by the application layer. Whether one provider is active or ten are running in parallel, the kernel guarantees that what goes in is exactly what comes out.

03

From zome
to omniverse

A single holon is powerful. But the real architecture emerges when holons nest. OASIS organises all data into a celestial hierarchy (a holarchy) where every level is both a self-contained whole and a part of something larger.

The STAR ODK maps this to a cosmic scale: Zomes group holons, Moons bundle Zomes into micro-apps, Planets unite Moons into feature domains, and Stars are full platform ecosystems. Apps grow through this hierarchy: a Moon earns its way to a Planet, and a Planet to a Star. Every level unlocks greater capability, visibility, and reach across the OASIS ecosystem.

Celestial mapping: Holon → Zome → Moon → Planet → Star → Solar System → Galaxy → Universe → Multiverse → Omniverse

Zomes & Holons

The atomic layer. Zomes group related holons (profiles, quests, NFTs, game items) into logical bundles. Each holon is the leaf node: one piece of data, stored on any provider.

Moons

Small, self-contained apps or micro-services. Every OAPP starts life as a Moon. Moons orbit their Planet parent and carry their own Zomes inside.

Planets

Major modules or feature domains within a Star. Each Planet is a holon, independently deployable with its own children, bound to its parent Star's identity.

Stars

Top-level platforms with their own ecosystems. A Star is an application that contains Planets and Moons as sub-modules. The root of an entire product.

04

One interface,
every provider

With OASIS, all providers come pre-integrated. It is as easy as a single call to hit your choice of blockchain or provider. No custom adapters, no integration tax, no vendor lock-in.

One Interface

IOASISStorageProvider defines CRUD for holons and avatars. Every provider implements it. Your app calls one API regardless of backend.

Auto-Failover

If the active provider goes down, OASIS switches to the next in the failover list. No downtime. No code changes.

Auto-Replication

Write to multiple providers simultaneously. Data lives on Ethereum and MongoDB at the same time. Redundancy by default.
